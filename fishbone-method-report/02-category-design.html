Element 02

大骨設計

魚骨図の大骨は、答えではなく 原因を並べるための分類軸 です。ASQ は Ishikawa の 6M を代表例として挙げつつ、利用者に合わせてカテゴリ名を工夫することも認めています。IHI も、原因がカテゴリで自然にまとまるなら classic fishbone、工程別に見たいなら process-type fishbone を使うとしています。

コツは、6Mを暗記して使うことではなく、「同じ原因を二重計上しにくい棚」をつくることです。ここで MECE の発想が効きます。

役割: 原因候補の棚づくり 基本参照: 6M / Process Fishbone 補助原理: MECE

大骨設計の基本パターン

6M型

Materials, Machinery, Methods, Measurement, Manpower, Mother Nature。製造や品質管理の初期設計に向きます。

プロセス型

IHI が勧める工程別の cause-and-effect。受付、審査、承認、引き継ぎのように、流れに沿って原因を置きます。

カスタム型

情報、ルール、UI、需要変動、依存先など、領域特有の切り口に置き換えます。サービスやソフトウェアではこちらが実務的です。

MECEをどう当てるか

Mutually Exclusive

各カテゴリが重なりにくいことです。McKinsey の Barbara Minto は、よいグルーピングは mutually exclusive and collectively exhaustive であるべきだと述べています。魚骨図に応用するなら、同じ原因が複数の大骨に入りにくい状態を目指します。

Collectively Exhaustive

重要な原因群が抜けにくいことです。完全網羅を保証するというより、抜けやすい論点を棚の段階で減らす実務上の工夫として使うのが現実的です。

ここでの MECE 適用は、魚骨図の公式ルールではなく、Barbara Minto の整理原則を分析設計に応用した解釈です。

大骨を選ぶコツ

TIP 1

責任単位ではなく因子単位で切る

「営業」「開発」「管理部」のような部署切りは、責任整理には便利でも原因整理には粗いことがあります。まずは方法、情報、環境、判断条件など因子で切ったほうが因果が見えます。

TIP 2

分類軸と原因語を混ぜない

大骨に「教育不足」「手順不備」を置くと、それ自体が原因になってしまいます。大骨は「人」「方法」「情報」のような棚、小骨に原因を書きます。

TIP 3

対象領域に合わせて6Mを崩す

ASQ もカテゴリ名の創意工夫を認めています。ソフトウェアや知識労働では「Measurement」を「可視化・モニタリング」に、「Machinery」を「ツール・UI」に置き換えたほうが使いやすいことがあります。

設計例

文脈 候補カテゴリ 狙い
製造 Materials / Machinery / Methods / Measurement / Manpower / Environment 従来の品質管理観点を漏らしにくい
サービス運用 人 / 手順 / 情報 / ツール / 負荷 / 外部条件 情報不足や繁閑差を扱いやすい
ソフトウェア運用 仕様 / 実装 / テスト / 監視 / デプロイ / 依存先 工程と技術要因を分けやすい
業務フロー改善 受付 / 判断 / 実行 / 引き継ぎ / 例外処理 process-type fishbone として使える

セルフチェック

大骨名は「棚」になっていて、原因そのものになっていないか。
同じ小骨が複数カテゴリに入りそうなら、カテゴリの切り方を見直せるか。
この問題領域に 6M が本当に合っているか。工程別のほうが自然ではないか。
カテゴリ名を見ただけで、参加者が何を書けばよいか想像できるか。
