Element 05

事実確認と検証

ここでの目的は、もっともらしい物語を 検証済みの原因 に絞ることです。Lean の解説では、current state を正しく理解し、問題が起きる場所をよく観察することが前提だとされています。IHI は root cause が複数ありうるとし、RCA2 は個人の失敗より システムの脆弱性 を見るべきだと強調しています。

コツは、枝の末端を「確定原因」と決める前に、観察、ログ、比較、再現、ヒアリングで確かめることです。確認できないものは仮説として扱います。

役割: 思い込みを削る 基本姿勢: blame ではなく system 成果物: 確認済み原因 / 未確認仮説

検証で見る4種類の材料

観察

問題が起きる現場、画面、手順、実物を見る。会議室の推論だけで閉じない。

データ

件数、率、時間、ログ、履歴で傾向を確認する。主観だけで枝を固定しない。

比較

起きるケースと起きないケース、月末と通常週、熟練者と非熟練者などを比べる。

再現

条件をそろえて再度起きるかを見る。消えるなら条件の切り分けができます。

検証の進め方

STEP 1

枝を仮説ラベル付きで並べる

まず「確認済み」「未確認」「反証あり」を分けます。ここを曖昧にすると、後で議論が混ざります。

STEP 2

問題が起きる場面へ戻る

LEI の説明どおり、current state と point of cause を見に行きます。実際の画面、手順、やりとりを見ると、中間事象が見つかることがあります。

STEP 3

差分で仮説を絞る

起きるケースと起きないケースを比べ、差がある条件だけを残します。複数 root cause の可能性を前提に、枝ごとに証拠の強さを見ます。

STEP 4

人ではなく条件で表現し直す

RCA2 の考え方に沿い、個人名や性格ではなく、どの条件が失敗を起こしやすくしたかへ書き換えます。

検証時のコツ

可視化

枝ごとに「根拠」を1行添えます。何を見てそう言うのかが曖昧だと、会議で強い人の意見が勝ちやすくなります。

反証

その原因が本当なら、起きないケースには何がないはずかを考えます。反証可能性を持たせると精度が上がります。

複数原因

一つに決め切れないときは無理に一本化せず、主要因と副要因の形で残します。複雑問題ではそのほうが現実的です。

避けたいパターン

記憶だけで確定する

「たしかいつもそうだった」で枝を決めると、例外条件や最近の変化を見落とします。

個人の失敗へ還元する

再発防止より責任整理に寄ります。条件、表示、タイミング、引き継ぎ、負荷に戻して書きます。

証拠の強さを区別しない

ログ確認済みの枝と、ヒアリング1件だけの枝を同列に扱うと判断を誤ります。

単独原因に決め打ちする

PSNet が指摘するように、複雑系では単一路線化が危険です。複数の contributing factors を保持します。
