Element 06

対策化とフォローアップ

分析の最後は、原因に対して どの条件を変えるか を決める工程です。IHI の RCA2 は、持続する改善と harm prevention を重視し、個人への注意喚起より システム改善 を重く見ています。ASQ は five hows を使うことで、解決策をより具体化できるとしています。

コツは、問題に直接対策を当てるのではなく、確認できた末端原因の真下へ対策を置くことです。原因と対策がずれると、実行しても効いたか分かりません。

役割: 再発防止へ落とし込む 重視点: system change / standardization 補助手法: Five Hows

対策化の基本構造

原因

例外条件が複数資料に分散し、入力時に必要項目を判断しにくい。

対策

入力画面に条件連動の必須表示を実装し、手順書側は例外条件一覧へのリンクに一本化する。

効果確認

月末3営業日の必須項目欠落率、差し戻し率、入力完了時間を継続測定する。

標準化

新手順を標準業務として文書化し、例外条件追加時の更新責任を定義する。

対策を作るときのコツ

TIP 1

強い対策を優先する

注意喚起、再教育、声かけだけでは再発しやすいです。表示、手順、入力制御、責任分担、監視など、条件そのものを変える対策を先に考えます。

TIP 2

Five Hows で実装粒度まで落とす

ASQ が紹介する five hows を使うと、「どうやって実装するか」をさらに分解できます。対策名だけで終わらず、誰が何をいつ変えるかまで書きます。

TIP 3

効果指標と運用指標を分ける

差し戻し率のような結果指標だけでなく、必須表示利用率、手順更新遅延件数のような先行指標も置くと追いやすくなります。

TIP 4

標準化まで含める

一度直っても、標準、責任、更新ルールがなければ戻ります。再発防止は implementation ではなく sustain まで含みます。

弱い対策になりやすい例

周知のみ

「気をつける」「共有する」だけで条件が変わらない。

教育のみ

手順やUIが悪いまま教育だけ追加し、負担を個人に返してしまう。

責任者不在

よい対策でも、誰が保守し続けるかが決まっていないと定着しない。
