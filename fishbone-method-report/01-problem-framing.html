Element 01

問題定義

このページは、`00. 前段の要素分析フィッシュボーン` の出力を、 トヨタ式なぜなぜ分析に入れる problem definition へ整形するためのページです。Lean Enterprise Institute は、5 Whys を適切に回す前提として specific problem definition と current state の正確な把握 を挙げています。つまり、ここではじめて「何を why するのか」を固定します。

コツは、00 で集めた枝をそのまま並べるのではなく、`current state`、`desired state / standard`、`gap`、`point of cause`、`impact` に圧縮することです。

役割: why の対象を固定する 入力元: 00 の要素分析フィッシュボーン 失敗しやすい点: 原因・対策の混入

問題定義で決めるべき6項目

1. Need / Priority

なぜこの問題を解く必要があるのかです。品質、納期、安全、顧客影響、手戻りなど、優先順位の理由を一行で置きます。

2. Current State

いま実際に何が起きているかを観察可能に書きます。数字、頻度、工程、場面を含めると安定します。

3. Desired State / Standard

本来どうあるべきかです。標準、通常状態、期待値、目標のどれを基準にするかを決めます。

4. Gap

current state と desired state の差です。差の大きさと、どの条件で差が開くかを書きます。

5. Point of Cause

どの工程・場面・条件から why を始めるかです。00 のフィッシュボーンで最も強い枝をここへ渡します。

6. Scope / Impact

どこまでを分析対象とし、何が困るのかです。対象外も書いておくと why が広がりすぎません。

00 から 01 へ圧縮する考え方

00 の枝で拾うもの 01 での書き換え先 例
起きる場面 / 起きない場面 発生条件、scope 月末3営業日、オンライン申請のみ
観察できる現象 current state 初回承認後の差し戻し率が18%
標準との差 desired state, gap 通常週は6%、月末は18%
もっとも強い枝 point of cause 承認前チェックが月末のみ後工程へ回る

実務で効くコツ

TIP 1

01 では「一文一問題」に圧縮する

00 で枝は複数あってよいですが、01 の problem statement はひとつの現象に絞ります。遅延とミスが同時にあるなら別 statement に分けたほうがよいです。

TIP 2

原因を problem statement に混ぜない

00 の枝で見えた strong hypothesis があっても、01 ではまだ正式な原因にしません。why を始めるための入口として point of cause に置くにとどめます。

TIP 3

標準を書かないと gap は書けない

current state だけでは why が拡散します。通常状態、標準手順、目標値など、何と比べてズレているのかを必ず添えます。

TIP 4

分析対象外も一緒に書く

「今回はオンライン申請だけ」「今回は月末3営業日だけ」のように対象外を決めると、why が全社論へ広がりにくくなります。

ありがちな失敗

枝の列挙をそのまま貼る

00 の魚骨図の枝を problem statement として並べてしまう。

標準不在

current state だけがあり、何との差かが書かれていない。

point of cause 不在

どの枝から why を始めるかが決まらず、分析が再び広がってしまう。
